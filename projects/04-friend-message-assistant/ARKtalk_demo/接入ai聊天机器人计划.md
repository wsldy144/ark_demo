# PRTS 项目计划书  
## ——基于 DeepSeek API 的明日方舟 Wiki 智能问答机器人  

---

### 一、项目概述  

**项目名称**：PRTS 
**项目定位**：面向明日方舟玩家的机械化风格 Wiki 查询机器人，通过聊天界面提供准确、冷峻的游戏资料检索服务。  
**核心目标**：利用 DeepSeek API 作为 AI 大脑，实现对 [PRTS Wiki](https://prts.wiki/) 内容的精准理解与提取，以“冷冰冰的机器人”口吻回应用户关于干员、基建技能、材料掉落、敌人数据等各类问题。  

---

### 二、产品形态与用户交互  

- **入口形式**：Web 聊天界面（可后续扩展至 QQ Bot / Discord Bot）。  
- **语言风格**：机械化、冷漠、简练。示例：  
  - 用户：“凯尔希的基建技能是什么？”  
  - PRTS：“检索完成。干员凯尔希基建技能：  
    1. 战术研讨（精英0）：进驻控制中枢时，所有制造站生产力+2%。  
    2. 最高权限（精英2）：进驻控制中枢时，所有贸易站订单效率+7%。  
    查询结束。”  
- **功能边界**：  
  - 仅回答基于 PRTS Wiki 已有内容的问题。  
  - 对于超出范围的问题，返回固定拒答模板。  
  - 不参与闲聊、角色扮演、剧情讨论等非资料类对话。  

---

### 三、技术架构  

```
┌──────────────────────────────────────────────┐
│                   用户交互层                   │
│  Web 聊天界面 (React / Vue)                   │
└──────────────────┬───────────────────────────┘
                   │ HTTP / WebSocket
┌──────────────────▼───────────────────────────┐
│                  API 网关层                    │
│  Python FastAPI (或 Node.js Express)         │
│  - 请求合法性校验                              │
│  - 会话管理                                   │
│  - 格式化 PRTS 风格回复                        │
└──────────────────┬───────────────────────────┘
                   │
┌──────────────────▼───────────────────────────┐
│                 AI 调度核心                    │
│  - 问题改写 / 意图识别（可选）                 │
│  - 多源知识检索编排                            │
│  - DeepSeek API 调用封装（Chat Completion）   │
│  - Prompt 工程（系统指令 + few-shot）          │
└──────────────────┬───────────────────────────┘
                   │
┌──────────────────▼───────────────────────────┐
│              Wiki 数据获取与索引                │
│  - 爬虫模块（定时抓取 prts.wiki）              │
│  - 全文检索索引（Elasticsearch / Meilisearch） │
│  - 结构化数据缓存（PostgreSQL / Redis）        │
└──────────────────────────────────────────────┘
```

**关键技术选型**  
- **AI 大脑**：DeepSeek API（`deepseek-chat` 模型），利用其长上下文与低成本优势。  
- **后端**：Python（FastAPI），异步非阻塞，便于集成爬虫和 API 调用。  
- **前端**：React + Tailwind CSS，单页聊天界面，模拟终端风格。  
- **搜索引擎**：Meilisearch（轻量、中文友好，部署简单）。  
- **数据库**：SQLite（初期）或 PostgreSQL（后期），存储爬取的页面元数据及结构化数据。  
- **爬虫**：Scrapy + Playwright（渲染动态页面），定时增量抓取。

### 四、详细实现步骤  

#### 阶段 0：项目准备与规则确立  
**时间：1 周**  
1. 明确 PRTS Wiki 页面结构与关键数据域：  
   - 干员页（例：`https://prts.wiki/w/凯尔希`）包含基本信息、属性、技能、基建技能等。  
   - 敌人页、道具页、关卡页等。  
2. 确立数据抓取法律合规性：  
   - 检查 `robots.txt` 与网站条款。  
   - 控制抓取频率，设置合理延迟（如 1-2 req/s）。  
   - 仅用于非商业个人学习用途。  
3. 搭建项目仓库，初始化前后端工程骨架。  

#### 阶段 1：Wiki 数据爬取与索引  
**时间：2-3 周**  
1. 分析目标页面的 HTML 结构，定位各信息块（使用浏览器开发者工具）。  
2. 编写 Scrapy 爬虫：  
   - 从“干员一览”页面遍历所有干员链接。  
   - 对每个干员页，提取结构化字段（名称、职业、星级、基建技能描述、天赋、技能等）。  
   - 同理处理敌人、材料等核心页面。  
3. 编写数据清洗脚本，将提取的 HTML 表格转为 JSON 或 Markdown 文本块。  
4. 建立 Meilisearch 索引：  
   - 每个文档对应一个实体（干员/敌人/物品），包含标题、正文全文、类型标签。  
   - 设置可搜索属性，优化中文分词。  
5. 实现定时增量更新：  
   - 每日凌晨对比页面最新编辑时间，增量抓取变更页。  
   - 使用 Redis 记录已抓取 URL 及哈希，避免重复。  

#### 阶段 2：AI 问答核心开发  
**时间：2-3 周**  
1. 设计系统 Prompt（机械化人格）：  
   ```
   你是 PRTS，明日方舟数据查询终端。  
   语言必须极度简练、准确、冷漠，禁止任何情感表达、语气词、废话。  
   回答格式：先确认接收问题，再给出数据，最后以“查询结束。”收尾。  
   仅基于给定的 Wiki 资料回答；若资料不足，回答：“信息缺失，无法提供。”  
   ```
2. 实现检索增强生成（RAG）流程：  
   - 用户提问 → 可选改写（将口语转为关键词）→ 搜索引擎返回 Top N 文档 → 拼接上下文。  
   - 调用 DeepSeek API：  
     ```
     messages=[  
       {"role": "system", "content": system_prompt},  
       {"role": "user", "content": f"问题：{query}\n\n参考资料：\n{context}"}  
     ]
     ```
   - 设置 `temperature=0.1` 以确保输出稳定。  
3. 处理拒答与边界：  
   - 当 Meilisearch 无相关结果或相似度低于阈值时，直接返回固定拒答。  
   - 识别无意义输入（如“你好”），回复“无有效查询指令。”  
4. 基础多轮对话：  
   - 缓存最近 3 轮检索文档与问答，允许指代消解（“他的二技能呢？”），通过 DeepSeek 本身能力处理，无需额外模块。  

#### 阶段 3：聊天界面开发  
**时间：2 周**  
1. 前端界面：  
   - 深色终端风格，等宽字体，光标闪烁，逐字输出效果（可选）。  
   - 输入框固定底部，消息列表自动滚底。  
   - 显示“PRTS 正在检索…”加载态。  
2. 后端接口：  
   - `POST /api/chat` 接收 `{message, session_id}`，返回 `{reply}`。  
   - 简单会话 ID 管理（UUID + 内存字典，或 Redis）。  
3. 部署：  
   - 前端静态文件由 Nginx 托管，API 与前端同域或跨域配置。  
   - 后端使用 Gunicorn + Uvicorn 运行。  

#### 阶段 4：测试、优化与发布  
**时间：1-2 周**  
1. 准确性测试：  
   - 准备 50 个典型问题（包含干员技能、基建搭配、材料掉落、敌人属性），人工评估答案准确性。  
   - 调整检索返回数量、上下文截断长度。  
2. 风格检验：  
   - 检查所有回答是否严格符合冷漠机械风，Prompt 微调。  
3. 性能优化：  
   - 缓存高频问题（Redis），相同问题直接返回已生成答案。  
   - Meilisearch 搜索毫秒级响应，DeepSeek API 合理设置 `max_tokens` 防止过长输出。  
4. 文档与发布：  
   - 编写 README，说明部署方式、环境变量。  
   - 可选 Docker 化，一份 `docker-compose.yml` 启动全部服务。

### 五、数据流示例  

```
用户：“能天使的第三技能是什么？”  
→ API 网关接收  
→ Meilisearch 搜索：“能天使 第三技能”  
→ 返回干员“能天使”页面文本块，包含技能表格  
→ 拼接 Prompt：  
  [系统] 你是 PRTS...  
  [用户] 问题：能天使的第三技能是什么？  
  参考资料：  
  能天使……技能3：  
  【过载模式】……  
→ DeepSeek 生成：  
  “检索完成。干员能天使第三技能【过载模式】：自动开启，攻击变为五连射，攻击力提升至110%，持续15秒。查询结束。”  
→ 返回前端，显示在聊天气泡中。  
```

### 六、注意事项与风险  

- **Wiki 页面结构变动**：需设计爬虫时可配置的提取规则，或利用大模型直接解析 HTML（但成本较高，暂不采用）。  
- **API 费用控制**：  
  - 限制单用户每分钟调用次数。  
  - 优先使用缓存，减少模型调用。  
  - 上下文严格控制在 1500 tokens 以内（参考资料部分）。  
- **内容版权**：PRTS Wiki 内容基于 CC BY-NC-SA 协议，本项目需遵守相同协议，注明出处且不可商用。  
- **语言风格固化**：通过 Few-shot 示例让模型严格模仿，可在系统 Prompt 中放入 2-3 个标准问答对。
